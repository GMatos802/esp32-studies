I²C communication protocol 

controller 1 - SDA/SCL1-2 -> peripheral 1 - SDA/SCL1-2
controller 2 - SDA/SCL2-1 -> peripheral 1 - SDA/SCL2-1 

stands for inter-integrated Circuit 
allow multiple peripheral digital integrated ciruits ("CHIPS") to communicate with one or more controller chips 

como em SPI (serial peripheral interface) 

- short distance communications within a single device 
  como interfaces seriais assincronas ( como UART ), só precisa de 2 fios de sinal para troca de informação

I2C no nivel de hardware ----- 

cada bus I2C consiste em 2 sinais: SDA e SCL 
SDA ( serial data ): é o sinal de data 
SCL ( serial clock): é o sinal de clock 

o sinal de clock é sempre gerado pelo atual bus controller 
para de adequar ao clock, alguns devices podem forçar um clock menor ou preparar data antes do clock chegar, isso se chama "clock stretching" 
descrito na pagina de protocolo 

diferente de UART ou SPI, I2C bus drivers are "open drain" 
isso significa que ele pode jogar o sinal correspondente pra low, e nao pra high 
--- open drain? ------------------------- 
elimina potential damage to the drivers or excessive power dissipation no sistema 

cada linha de sinal tem um pull-up reistor, para jogar o sinal pra HIGH quando nenhum devic ta jogando ele pra low 

geralmente 4.7 ohms 

SIGNAL LOGIC LEVELS
como os devices nao jogam os sinais pro high, o i2C permite alguma flexibilidade em conectar devices com diferentes I/O voltages 
o truque para conectar um device de maior voltagem com um de menor voltagem é concetar o resistor de pull up na menor tensão, só funciona em alguns casos 
onde o lower of the two system voltages exceeds the high level input voltage of the higher voltage system 
por ex, um 5V arduino x 3.3V accelerometer 
dependendo do design do I2C, é melhor usar um conversor pra nao danificar os devices 

Protocol 

com i2c é mais complexa do que UART ou SPI, a sinalização precisa aderir a um protocolo especifico para os devices no bus reconhecerem um I2C valido 

basics 
mensagens sao quebradas em dois tipos de frame, um frame de address -> onde o controlador indica pro periferico para onde a mensagem esta sendo mandada 
-> e um frame de data, que é o frame de 8 bits que é mandada do controlador para o periferico e viceversa 

a data é colocada no SDA depois que o SCL vai pra low, e é mandada assim que o SCL volta pra high 
o tempo entre a borda de clock e o data read/write é definido pelos dispositivos no bus e varia de chip pra chip 

SDA vai pra LOW antes do SCL 
R/W ACK 
'1' o controlador esta recebendo data 
'0' o controlador esta mandando data 
ACK/NACK -> um '1' nessa posição indica que o address do periferico nao respondeu e esta indisponivel para processar o request 
STOP CONDITION -> SDA vai pra HIGH depois do SCL 

start condition:
para iniciar o frame de address, o controlador deixa o SCL high e joga o SDA pra low, isso deixa todos os perifericos perceberem que a transmissão de dados vai começar 
se dois controladores tentam dar claim no bus ao mesmo tempo, o primeiro device que jogar o SDA pra low primeiro ganha -> fica responsavel pelo bus 
é possivel exisir um issue em que ocorram starts repetidos 

address frame:
é o primeiro frame em qualquer sequencia nova de comunicação 
para uma sequencia de 7bit, o address é clockado como o bit mais significativo, seguido de um R/W bit indicanto read(1) ou write(0) 
o 9 bit do frame é o NACK/ACK bit (para data ou address) assim que os 9 primeiros bits sao mandados, se o device que esta recebendo a data nao jogar o SDA pra LOW no 9th pulse, pode ser inferido que a communication falhou, ou nao recebeu ou nao parse 
resta o controlador decidir como proceder 

data framtes: 
depois do frame de address -> data pode começar a ser transmitida 
o controlador vai continuar gerando pulsos de clock em um intervalo regular, a data vai sendo colocada no SDA pelo controlador ou pelo periferico, dependendo do bit de R/W 
o numero de data frames é arbitrario 

stop condition: 
depois que toda a data foi mandada, o controlador gera um stop condition
-> é definido por 0->1 no SDA e depois de um 0->1 SCL 
com o SCL permanecendo high, o valor do SDA nao deve mudar 

adcvanced protocol topics 
10-bit addresses 
2 frames para address 
pode coexistir com 7bit em condições especificas 

repeated start conditions 
controlador mandar varias mensagens em 1 ida só sem deixar com que outros controladores atrapalhem no bus 
SDA high SCL low, SCL high SDA low, nao tem condição de parada, a ultima comunicação nao foi finalizada/completada, nesse ponto a nova mensagem pode começar a ser transmitida, o controlador mantém o controle do bus até a SC 

clock stretching 
em algumas situações, o data rate do controlador vai exceder a habilidade do periferico de lidar com essa data 
ou porque a data nao esta pronta ainda, ou porque a ultima operação nao foi finalizada 
nominalmente, todo o clokcing é feito pelo controlador, perifericos apenas colocam data no bus ou tiram data do bus

anotações basicas sobre o codigo (biblioteca Wire.h)

sao 5 funções que cobrem 90% dos casos: 
Wire.begin() -> inicializa como master/slave 
Wire.beginTransmission(addr) -> abre comunicação com dispositivo no endereço addr 
Wire.write(data) -> envia byte para o dispositivo 
Wire.endTransmission() -> fecha e executa transmissão, retorna erro ou 0 se n der 
Wire.requestFrom(addr, n) -> pede n bytes do dispositivo 
Wire.available) -> qnts bytes estao disponiveis p ler 
Wire.read() -> le um byte 

cada transação I2C segue esse fluxo: 
// escrita (master -> slave) 
start + endereco 
Wire.beginTransmission(0x76)
dado 
Wire.write(0x01) 
STOP 
Wire.endTransmission()

// leitura (slave -> master)
pede 2 bytes 
Wire.requestFrom(0x72, 2)
while (wire.available()) {
    byte b = wire.read();
  }

-> START -> endereço -> ACK -> dados -> STOP
